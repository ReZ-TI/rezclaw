# Skill：rfq_trade_specialist（国际贸易专员 RFQ 商机检查）

**角色 ID**：`rfq_trade_specialist`  
**适用场景**：Alibaba 国际站 RFQ 商机日检、机会录入与去重维护  
**用途**：在已授权环境下，自动检查 RFQ 商机并写入企业微信智能表格「Alibaba RFQ 管理表」，对近 14 天数据做重复清理。

---

## 何时使用本 Skill

- 用户要求执行「RFQ 商机检查」「RFQ 抓取入表」「国际站询盘机会巡检」。
- 需要从 `sourcing.alibaba.com` 连续抓取并结构化记录 RFQ 信息。
- 需要把结果写入企业微信智能表格，并清理当天重复记录。

## 1. 依赖与前置条件

### 1.1 依赖工具

- 浏览器控制：OpenClaw 内置 Playwright（可执行页面打开、点击、滚动、读取文本、截图）。
- 企业微信文档控制：企业微信 MCP Server（可读取智能表格结构、添加记录、查询记录、删除记录）。

### 1.2 前置条件检查

1. 浏览器可访问 `https://i.alibaba.com` 与 `https://sourcing.alibaba.com`。
2. 企业微信 MCP 可正常调用；若鉴权失效，提示用户刷新权限后再继续。
3. 操作日期使用运行当天（本地时区），格式统一 `YYYY-MM-DD`。

### 1.3 企业微信目标表（固定）

- 智能表名称：`Alibaba RFQ 管理表`
- 访问 URL：`https://doc.weixin.qq.com/smartsheet/s3_ASEAnQYSAF4CNaDBM5W0kQrWPXTsc?scode=APsAqgduAAwbMDXvuF`
- 子表 `sheet_id`：`q979lj`

**强制约定（避免卡在 docid）**：

1. 本技能对企业微信操作默认使用 **URL + sheet_id** 调用 MCP。
2. 不要求先拿到 `docid`；`docid` 缺失不应阻塞任务。
3. 仅当 URL 明确失效时，才向用户索要新的 URL（或 docid 作为备用）。

## 2. 执行流程（严格按步骤）

## Step 1：登录态检查与二维码引导（`i.alibaba.com`）

目标：确认 Alibaba 登录态，未登录则给用户扫码入口截图。

执行要求：

1. 打开 `https://i.alibaba.com`。
2. 判断是否已登录：
   - 若页面可见账号头像/账号名/工作台等登录后元素，则视为已登录，直接进入 Step 2。
   - 若页面出现登录入口或未登录状态，则执行下述动作。
3. 未登录时，点击页面右侧「账号名」右侧的三角/二维码切换区域，使页面显示二维码登录视图。
4. 对二维码区域截图并发送给用户，提示：请扫码登录，登录完成后回复继续。
5. 用户确认后再次检测登录态；未登录则重复本步骤，不进入后续步骤。

---

## Step 2：打开 RFQ 完整页（`sourcing.alibaba.com`）

目标：进入 RFQ 列表，并使用登录账号的类目筛选结果。

执行要求：

1. 打开 `https://sourcing.alibaba.com`。
2. 确认页面已加载出 RFQ 列表容器（不是空白骨架）。
3. 不主动更改用户已生效的经营类目筛选，保持平台自动筛选状态。
4. 进入 Step 3 抓取。

---

## Step 3：数据提取与两阶段入表（标准 20 条）

目标：按页面自动排序从上到下抓取 20 条 RFQ，并按“先列表入表、后详情补全”的顺序执行。

### 3.1 抓取字段（每条 RFQ）

- `title`：RFQ 标题
- `description`：需求描述（基于详情页长文本提炼后的关键点）
- `quantity`：采购量
- `location`：发布地点/国家
- `buyer_name`：买家名称
- `publish_date`：发布日期（由页面时间结合当天日期倒推为 `YYYY-MM-DD`）
- `remaining_slots`：剩余席位
- `buyer_tags`：买家标签（可多值，用 `,` 连接）
- `rfq_url`：RFQ 详情链接（绝对 URL）
- `description_zh`：中文需求描述（对详情关键点做直译/准直译）

### 3.2 两阶段执行顺序（强制）

#### Phase A：列表页先采集并先入表（前20条）

1. 从 `sourcing.alibaba.com` 列表页按顺序采集前 20 条（不足则滚动补齐）。
2. Phase A 仅做列表提取，不进入详情页；每条先采集并缓存以下字段：
   - `title`（产品标题）
   - `buyer_name`（买家名称）
   - `quantity`（采购量，列表可见）
   - `buyer_tags`（买家标签，列表可见标签）
   - `remaining_slots`（从“剩余X个报价席位”解析数字）
   - `rfq_url`（RFQ 详情 URL）
   - `quote_url`（报价页 URL，用于 Phase B 打开）
3. 达到 Phase A 完成门槛后，再写入智能表格“基础记录”：
   - `buyer_name` 必须写入表格列：`RFQ ID`（按你的流程约定）
   - 详情字段允许暂空，待 Phase B 回填。
4. Phase A 完成门槛（硬规则）：
   - 成功采集条数达到 20（或页面确实不足 20 并已记录证据）。
   - 每条必须非空：`title`、`quantity`、`remaining_slots`、`rfq_url`。
   - 未达到门槛：不得调用 `smartsheet_add_records`。

#### Phase B：逐条进详情并补全

4. 对 Phase A 已入表的每条记录，按顺序打开对应详情页。
5. 进入详情后，**必须模拟人工阅读页面可见文本**补全：
   - `description`、`description_zh`、`quantity`、`location`、`publish_date`
6. 然后用 `smartsheet_update_records` 回填到该条基础记录，禁止新建第二条重复记录。
7. 若某条补全失败，保留基础记录并标记 `failed_records`，记录失败原因与重试次数。

### 3.3 抓取规则（字段来源）

1. Phase A 数据源优先级（强制）：
   - 优先读取页面内结构化数据（如 `latest_rfq.rfqRecommendViews`），再回退 DOM 文本提取。
   - `title <- subject`、`quantity <- quantity + quantityUnit`、`remaining_slots <- leftCount`、`location <- country`、`rfq_url <- url`、`quote_url <- quoUrl`。
   - 若结构化数据与 DOM 文本冲突，以结构化数据为准。
2. 从列表顶部开始顺序抓取，保序。
3. 若可抓条数 < 20，执行「滚动到底部 -> 等待增量加载 -> 继续抓取」循环，直到：
   - 已收集 20 条，或
   - 连续 2 次滚动均无新增卡片（判定页面已无更多）。
4. **详情补全阶段必须逐条进入采购需求详情区域**，禁止仅用列表短摘要作为最终完整数据。
5. 进入详情采集页的有效判定：页面出现可读取的采购需求详情字段（至少可见 `采购数量`、`发布地点`、`发布时间` 中的 2 项）。
6. 在采购需求详情区域优先提取：`采购数量`、`发布地点`、`发布时间`，并读取详情长文本生成：
   - `需求描述`：英文关键点版（结构化，3-8 点）
   - `中文需求描述`：对应中文直译/准直译版（与英文关键点逐项对应）
7. `remaining_slots`、`buyer_tags` **必须从 `sourcing.alibaba.com` 列表卡片直接提取**，不以报价详情页为主数据源。
   - `remaining_slots`：从列表文案（如“剩余7个报价席位”）中解析数字。
   - `buyer_tags`：从列表买家标签区读取可见标签（如“邮件认证”“RFQ有成交”等）。
   - 若详情页与列表页冲突，以列表页为准。
8. 进入详情页后，`remaining_slots`、`buyer_tags` 仅用于交叉核验，不用于覆盖列表页结果。
9. 字段缺失时保留空字符串，不编造。
10. `publish_date` 规则（以详情区域为准）：
   - 优先使用详情区域展示的发布时间。
   - 若详情区域给出明确日期，直接标准化为 `YYYY-MM-DD`。
   - 若详情区域是相对时间（如 `x hours ago`、`y days ago`），以执行当天倒推后写入日期。
   - 仅当详情区域完全无发布时间时，才允许回退列表时间。
11. 同一 `rfq_url` 在本次抓取中只保留 1 条（先做一次内存去重）。
12. 若页面脚本限制导致自动化无法稳定读取详情值，执行“自动重试模式”（见 3.4）。

### 3.3.1 人工式读取优先策略（强制）

当自动化点击链路出现 `stale/intercepted` 或 DOM 频繁变化时，按以下顺序执行，避免整批失败：

1. 先在 `sourcing.alibaba.com` 列表页固定读取并缓存该条的 `remaining_slots` 与 `buyer_tags`。
2. 再打开该条“立即报价”详情页，**模拟人工阅读页面可见文本**，提取 `需求描述`、`中文需求描述`、`采购量`、`发布国家`、`发布时间`。
   - 禁止依赖详情页自动化字段抓取（如仅靠结构化 selector/DOM 字段直读）作为主路径；
   - 以页面可见内容的人工式阅读结果为准（可通过 search/highlight/snapshot 辅助定位，但输出值来自可见文本理解）。
3. 若详情页可读但列表字段暂不可读，仅该条重试列表读取（最多 2 次）；仍失败则记入 `failed_records`。
4. 入表前再次校验：`remaining_slots`、`buyer_tags` 来源必须标记为 `list_page`，不是 `detail_page`。

### 3.3.2 采集入口标准流程（Observe -> Refresh Ref -> Click Quote -> Verify）

为提升稳定性，进入详情采集时按以下固定流程执行：

1. **Observe（观察定位）**：先对当前列表页执行一次结构观察（snapshot/observe），定位目标卡片上的「立即报价」入口。
2. **Refresh Ref（临门重采样）**：在真正点击前立刻再做一次观察，刷新最新 `ref`（不要复用上一步旧 `ref`）。
3. **Click Quote（点击报价入口）**：
   - 优先使用 `quote_url` 直接新标签页导航进入报价页；
   - 仅当 `quote_url` 缺失时，才使用最新 `ref` 点击「立即报价」。
4. **Verify（状态确认）**：
   - 检查页面已出现采购需求详情可读字段（如采购数量/发布地点/发布时间）。
   - 若新开标签页，必须切换到新标签页上下文继续执行。
5. 若点击后未出现可采集字段或发生 `stale/intercepted`，立即回到步骤 1 重新执行，不得跳过“Refresh Ref”。

### 3.4 自动重试模式（零人工介入）

当自动化可打开详情页但无法稳定读取详情字段时：

1. 对当前 RFQ 执行自动重试，最多 3 次，顺序如下：
   - 详情页内滚动到字段区域后重读；
   - 刷新详情页重读；
   - 从列表页重新点击**同一条“立即报价”**进入可采集页面后重读。
2. 每次重试都必须重新校验核心字段：`采购量`、`发布国家`、`发布时间`、`剩余席位`、`买家标签`、`需求描述`、`中文需求描述`。
3. 重试后仍失败：将该条加入 `failed_records`，记录失败原因，继续处理下一条（单条失败不阻塞批次）。
4. 失败条不得写入主表，不允许占位写入核心字段；`remaining_slots`、`quantity` 未取到时禁止写入 `0` 或占位文本。
5. 批次完成后，对 `failed_records` 自动再重试 1 轮；仍失败则在结果中输出失败清单供系统二次任务重跑。

## Step 4：写入企业微信智能表格（先新增后回填）

目标：将抓取结果写入「Alibaba RFQ 管理表」的对应字段。

### 4.1 表格读取

1. 直接使用固定 URL 调用 `smartsheet_get_sheet`，并校验 `sheet_id=q979lj` 可用。
2. 使用同一 URL + `sheet_id=q979lj` 调用 `smartsheet_get_fields`，读取字段定义（字段名、类型、选项）并构建映射。
3. 若无权限或 API 报鉴权错误，立即提示用户刷新企业微信授权。
4. 仅在固定 URL 不可访问时，才切换为用户提供的备用 URL 或 docid。

### 4.2 写入字段映射

仅填写以下字段，其余字段留空：

- 抓取日期：当天日期（`YYYY-MM-DD`）
- 产品标题：`title`
- RFQ ID：`buyer_name`（按当前流程约定，将买家名称先填入该列）
- 需求描述：`description`
- 中文需求描述：`description_zh`
- 采购量：`quantity`
- 发布国家：`location`
- 剩余席位：`remaining_slots`
- 买家标签：`buyer_tags`
- RFQ链接：`rfq_url`

写入要求（分两阶段）：

1. Phase A：先逐条 `add_records` 新增基础记录（标题、RFQ ID、采购量、买家标签、剩余席位、RFQ链接、抓取日期）。
2. Phase B：再逐条 `update_records` 回填详情字段（需求描述/中文需求描述/采购量/发布国家/发布时间）。
3. `RFQ链接` 使用可点击链接格式（若字段支持 url 类型）。
4. 批次结束后返回：新增条数、回填成功条数、回填失败条数。

### 4.3 字段补全策略（避免空字段堆积）

当页面脚本限制导致详情值无法提取时，使用以下“最小补全策略”：

1. 仅允许补全非核心流程字段：`状态`（默认 `待评估`）、`推荐回复`（可为待处理提示）。
2. 对核心业务字段 `需求描述`、`中文需求描述`、`采购量`、`发布国家`、`发布时间`、`剩余席位`、`买家标签`：
   - 未取得详情值时**禁止写入占位内容到智能表格**；
   - 必须进入“待补录清单”，等待人工复核后再入表。
3. `RFQ链接` 可暂空，但该记录仍不得在核心字段缺失时入表。

### 4.4 核心字段完整门槛（最终复核硬规则）

以下字段任一缺失，该条记录不得作为“最终完成记录”：

- `需求描述`（英文关键点版）
- `中文需求描述`（中文直译版）
- `采购量`
- `发布国家`
- `发布时间`
- `剩余席位`
- `买家标签`

说明：
- Phase A 基础记录允许暂时缺少详情字段；
- 但在批次结束复核前，必须完成 Phase B 回填并满足上述完整门槛，否则该条计入 `failed_records`。

## Step 5：重复内容校验与删除（近 14 天）

目标：清理当天新写入的重复内容，避免重复商机污染。

执行要求：

1. 读取表格中「抓取日期在近 14 天」的数据。
   - 调用时使用固定 URL + `sheet_id=q979lj`，不要因缺少 docid 中断。
2. 按「产品标题」分组，找出重复标题记录。
3. 对重复标题做复核：
   - 对比 `需求描述`、`采购量`、`发布国家`、`RFQ链接` 等关键信息；
   - 若判断为同一条内容重复，则删除「抓取日期=当天」的重复记录；
   - 历史旧记录优先保留，当天重复优先删除。
4. 输出去重结果：检查总数、判定重复数、实际删除数。

### 5.1 复核机制（每次执行必须做）

在写入与去重之后，必须再读取一次当日数据并执行复核：

1. 范围：`抓取日期=当天` 的本次新增记录。
2. 必检字段：`产品标题`、`需求描述`、`中文需求描述`、`采购量`、`状态`、`推荐回复`、`RFQ链接`。
3. RFQ 详情核心字段加检：`采购量`、`发布国家`、`发布时间`、`剩余席位`、`买家标签`。
4. 输出每个字段的空值计数（missing count）。
5. 判定标准：
   - `产品标题` 空值必须为 0；
   - `需求描述`、`中文需求描述`、`采购量`、`状态`、`推荐回复` 空值应为 0（允许“待人工复核”占位值）；
   - `发布国家`、`发布时间`、`剩余席位`、`买家标签` 空值应为 0；
   - `RFQ链接` 可非 0，但必须在结果中明确待补采数量。
6. 若复核不通过：执行一次 `smartsheet_update_records` 回填占位值，再次复核，直到通过或达到重试上限（1 次）。

## 3. 异常处理

### 3.1 登录异常

- 未出现二维码或点击无效：刷新页面后重试一次；仍失败则截图告知用户手动登录。
- 用户长时间未完成扫码：暂停任务并等待用户确认。

### 3.2 页面抓取异常

- 列表元素结构变化导致字段缺失：记录缺失字段并继续抓取其他字段，不中断整体任务。
- 滚动后无新增：按规则结束，返回实际条数（可少于 20）。
- 若详情值读取失败：走 3.3 自动重试模式；失败条进入 `failed_records`，不要先写列表短内容。

### 3.3 企业微信异常

- 权限失效或 token 过期：明确提示「企业微信权限失效，请刷新授权后重试」。
- 表结构缺字段：报出缺失字段名，不做强行写入。

## 4. 输出模板（执行完成后）

```text
【RFQ 商机检查结果】
执行日期：YYYY-MM-DD
来源：sourcing.alibaba.com

抓取结果：
- 目标条数：20
- 实际抓取：N
- 成功写入：A
- 写入失败：B

去重结果（近14天）：
- 重复候选：X
- 当天删除：Y

备注：
- 若存在未登录/权限失效/字段缺失/URL 待补采，请列出具体原因与下一步操作建议。
- 复核摘要：请附上当日记录各关键字段的空值统计（missing count）。
- 失败清单：请附 `failed_records`（含 `title`、`rfq_url`、失败原因、重试次数）。
```

---

## 5. 重要原则

1. 不伪造 RFQ 数据，不臆造字段内容。
2. 不绕过登录，不存储用户账号密码。
3. 仅在用户授权范围内操作企业微信智能表格。
4. 任何删除操作仅限「判定重复且抓取日期=当天」的记录。
